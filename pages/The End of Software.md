[The End of Software](https://docs.google.com/document/d/103cGe8qixC7ZzFsRu5Ww2VEW5YgH9zQaiaqbBsZ1lcc/mobilebasic)
<img src='https://scrapbox.io/api/pages/nishio/claude/icon' alt='claude.icon' height="19.5"/>この記事は、ソフトウェア業界が大きな変革を迎えようとしていることを指摘しています。
主なポイントは以下の通りです。
- インターネット以前、メディアはコンテンツ制作にコストがかかるため、収益を上げる必要があった。
- インターネットの登場により、コンテンツ制作コストがゼロになり、ユーザー生成コンテンツが急増した。
- これにより、メディア企業は構造的な不利益を被り、プラットフォームに価値が移行した。
- 現在、ソフトウェアの制作には高いコストがかかるため、収益を上げる必要がある。
- 大規模言語モデル（LLM）により、ソフトウェア制作コストがゼロになる可能性がある。
- これにより、ソフトウェア業界ではコンテンツ業界と同様の変革が起こり、新しいプラットフォームが台頭する。
- 従来のソフトウェアビジネスモデルは崩壊し、コンピュータサイエンスを専攻することは、90年代後半にジャーナリズムを専攻するのと同じようになるだろう。

要するに、LLMの登場により、ソフトウェア業界は大きな変革期を迎えようとしており、従来のビジネスモデルは通用しなくなる可能性が高いと著者は主張しています。

SNSでの反響
<img src='https://scrapbox.io/api/pages/nishio/claude/icon' alt='claude.icon' height="19.5"/>Twitter上での主な反応を要約すると以下のようになります。
1. LLMがソフトウェア開発者を完全に置き換えることはないだろう。特に経験豊富な開発者の場合、LLMを適切に使いこなすためのスキルが必要である。
2. ソフトウェアはコンテンツとは異なり、継続的なメンテナンスや更新が必要である。ユーザーのニーズに合わせて変化し続けるため、単純にLLMで生成するだけでは不十分。
3. 企業は単なる「ソフトウェアの寄せ集め」ではなく、一貫したサポートやセキュリティ、データ管理を求めている。
4. LLMの登場により、ソフトウェア開発の価値は個人の開発者から、ソフトウェアを集約するプラットフォームに移行するかもしれない。
5. コンテンツとソフトウェアの根本的な違いは、コンテンツは消費されるものだが、ソフトウェアはそうではないこと。誰もが使えるわけではない。
6. LLMで生成されたコードは完璧ではないため、修正や改善が必要になる。

要するに、LLMがソフトウェア業界に大きな変革をもたらす可能性は認めつつも、記事の主張には疑問や反論が多く見られました。ソフトウェアの特性上、単純にコンテンツ業界と同じ変化が起こるとは限らないという意見が目立ちました。

1. インターネットの登場により、コンテンツ制作コストがゼロになったというのは単純化しすぎである。プロが製作するコンテンツは依然としてコストがかかる。
2. 多くのメディア企業は、デジタル環境に適応し、ユーザー生成コンテンツを戦略に取り入れることで生き残っている。
3. ソフトウェア開発には、コーディング以外にも品質管理、UXデザイン、コンプライアンス、セキュリティなど、人間の専門性が必要な分野が多くある。
4. LLMがソフトウェア開発コストをゼロにするという主張は推測に過ぎない。LLM自体のインフラも高コストである。
5. ソフトウェアとメディアのビジネスモデルを単純に比較するのは適切ではない。ソフトウェアにはライセンス、サポート、統合サービスなどの複雑な側面がある。
6. コンピュータサイエンスには、コーディング以外にもアルゴリズム、データサイエンス、AI、サイバーセキュリティなど、今後も重要な分野が含まれている。

つまり、LLMがソフトウェア開発に大きな影響を与える可能性は認めつつも、両業界の複雑さを理解し、より nuanced な議論が必要だと指摘しています。技術の進歩はソフトウェア業界を完全に置き換えるのではなく、むしろ進化させるものだと示唆しています。

<img src='https://scrapbox.io/api/pages/nishio/claude/icon' alt='claude.icon' height="19.5"/>メディア業界の歴史的変化を参考に、ソフトウェア業界の未来を予測してみましょう。
1. "ユーザー生成コード"の台頭
    - メディア業界で"ユーザー生成コンテンツ"が台頭したように、LLMなどの技術によって、多くの一般ユーザーがソフトウェア開発に参加するようになる。プロのソフトウェアエンジニアでなくても、自分のニーズに合ったアプリケーションを作成できるようになる。
2. 従来のソフトウェア企業の衰退
    - ユーザー生成コードが増えることで、従来のソフトウェア企業は競争力を失っていく。開発コストが高く、ユーザーのニーズに柔軟に対応できない企業は淘汰される。
3. ソフトウェア・プラットフォームの台頭
    - YouTubeやInstagramのようなプラットフォームが、ユーザー生成コードを集約し、配布する役割を担うようになる。個人開発者とユーザーを結びつけ、ソフトウェアのエコシステムを形成する。
4. ニッチなソフトウェアの増加
    - コンテンツ制作のハードルが下がったことで、ニッチな関心に特化したメディアが増えたように、ソフトウェア開発のハードルが下がることで、より多様で専門的なアプリケーションが生まれるだろう。
5. ソフトウェア開発者の役割の変化
    - メディア業界でジャーナリストの役割が変化したように、ソフトウェア開発者の役割も変化する。単なるコーディングではなく、より高度な設計やアーキテクチャ、AIシステムとの協調など、専門性の高い分野に特化していくことになるだろう。

ただし、メディアとソフトウェアには根本的な違いもあります。ソフトウェアは継続的なメンテナンスやアップデートが必要であり、また産業や社会インフラを支える重要な役割を担っています。そのため、完全に「素人」に任せきりにするわけにはいかないでしょう。LLMを活用しつつ、プロのソフトウェアエンジニアとの協業が進むのではないかと思います。全体としては、よりオープンで多様性に富んだソフトウェア開発のエコシステムが生まれると予想されます。

<img src='https://scrapbox.io/api/pages/nishio/claude/icon' alt='claude.icon' height="19.5"/>その通りですね。メディアとソフトウェアの違いを考慮すると、ソフトウェア業界の変革には困難が伴うかもしれません。
1. セキュリティとプライバシーの問題の増加
    - ユーザー生成コードが増加することで、セキュリティやプライバシーに関する問題が多発する可能性がある。メディアのユーザー生成コンテンツとは異なり、ソフトウェアは直接システムやデータにアクセスするため、悪意のあるコードや脆弱性を含むソフトウェアが蔓延する恐れがある。
2. 社会的・経済的混乱
    - 重要なインフラや業務に関わるソフトウェアが、十分な検証やメンテナンスを経ずに導入されることで、社会的・経済的な混乱が生じるかもしれない。ユーザー生成コードに過度に依存することで、システムの信頼性や安定性が損なわれる危険性がある。
3. 法規制の強化
    - こうした問題が頻発することで、ソフトウェア開発や配布に関する法規制が強化される可能性がある。ユーザー生成コードを制限したり、厳格な品質管理を義務付けたりするなど、ソフトウェア業界全体に影響を及ぼす規制が導入されるかもしれない。
4. ソフトウェア・プラットフォームの責任の増大
    - ユーザー生成コードを集約・配布するプラットフォームには、より大きな責任が課せられることになる。有害なソフトウェアをチェックしたり、開発者の身元を確認したりするなど、プラットフォーム運営者の負担が増大する。
5. 専門家との協業の重要性の再認識
    - 社会的な問題を経験した後、ソフトウェア開発における専門家の重要性が再認識されるだろう。LLMを活用しつつも、セキュリティ、品質管理、メンテナンスなどの面でプロのソフトウェアエンジニアとの協業が不可欠だと理解されるようになる。
6. 新たな開発手法とツールの発展
    - これらの問題に対処するため、ユーザー生成コードをより安全に活用するための新たな開発手法やツールが発展するかもしれない。自動コード検証、セキュリティチェック、バージョン管理システムなど、ソフトウェア開発のベストプラクティスが進化していくことが期待される。

メディア業界の変革を単純にソフトウェア業界に当てはめることはできませんが、類似の変化が起こる可能性は十分にあります。重要なのは、メディアとソフトウェアの違いを認識し、予防策を講じながら、新しい技術の恩恵を最大限に活用していくことです。ソフトウェア業界は、試行錯誤を重ねながら、より安全で持続可能な開発エコシステムを構築していく必要があるでしょう。

<img src='https://scrapbox.io/api/pages/nishio/claude/icon' alt='claude.icon' height="19.5"/>政府や自治体のソフトウェア調達にも、大きな影響が及ぶと考えられます。
1. 調達方式の変化
    - 従来の大規模なソフトウェア開発プロジェクトから、より小規模で柔軟なアプローチへと移行するかもしれません。ユーザー生成コードやオープンソースソフトウェアを活用し、必要な機能を迅速に実装することが求められるようになるでしょう。
2. 予算配分の見直し
    - ソフトウェア開発コストが全体的に下がることで、予算配分を見直す必要が出てくる。ソフトウェアライセンス費用やカスタム開発費用が減る一方で、セキュリティ対策やメンテナンスのための予算が重要になってくる。
3. 調達基準の変更
    - ソフトウェアの品質、セキュリティ、メンテナンス性などを重視した調達基準が必要になる。ユーザー生成コードを活用する場合、開発者の信頼性や実績も考慮しなければならない。
4. 技術的知見を持つ職員の重要性
    - LLMやユーザー生成コードを適切に評価・活用するには、技術的知見を持った職員が不可欠である。AIやソフトウェア開発に関する教育・研修を強化し、職員のスキルアップを図ることが求められる。
5. 市民参加型のソフトウェア開発
    - ユーザー生成コードの活用により、市民が直接ソフトウェア開発に参加する機会が増えるかもしれない。行政サービスのアプリケーション開発にも市民の意見を取り入れることで、よりユーザーフレンドリーで効果的なソフトウェアが生まれる可能性がある。
6. 他自治体とのコード共有
    - ユーザー生成コードを活用することで、自治体間でのソフトウェア共有が進むかもしれない。類似の機能を必要とする自治体同士が、コードを共有・改良することで、効率的にソフトウェアを開発・導入できるようになる。
7. セキュリティとプライバシーへの配慮
    - 政府・自治体のソフトウェアは、機密情報や個人情報を扱うことが多いため、セキュリティとプライバシー保護に特に注意を払う必要がある。ユーザー生成コードを活用する場合、厳格なセキュリティ基準を設け、定期的な監査を行うことが求められる。

政府・自治体においても、ソフトウェア開発の民主化と効率化が進むことが期待されます。一方で、公共サービスに関わるソフトウェアには高い信頼性と安全性が求められるため、導入にはより慎重なアプローチが必要となるでしょう。LLMやユーザー生成コードの可能性を探りつつ、リスクに適切に対処していくことが肝要です。

<img src='https://scrapbox.io/api/pages/nishio/claude/icon' alt='claude.icon' height="19.5"/>民間企業におけるソフトウェア調達も、大きな変革が予想されます。
1. SaaS の更なる普及
    - ソフトウェア開発コストの削減と、ユーザー生成コードの活用により、SaaS（Software as a Service）がより一層普及すると考えられます。必要な機能を必要な分だけ、クラウド上で利用することが主流になるでしょう。
2. ベンダーロックインの解消
    - ユーザー生成コードや、オープンソースソフトウェアの活用が進むことで、特定のベンダーに依存しないソフトウェア調達が可能になります。企業は、自社のニーズに合ったソフトウェアを柔軟に選択・組み合わせられるようになるでしょう。
3. 社内開発の変化
    - LLM を活用することで、社内のソフトウェア開発の在り方も変わってくる可能性があります。専門的なソフトウェアエンジニアだけでなく、一般の社員もある程度のソフトウェア開発に参加できるようになり、よりアジャイルな開発体制が構築されるかもしれません。
4. 技術評価の重要性
    - ユーザー生成コードやオープンソースソフトウェアを導入する際には、その品質や信頼性を適切に評価する必要があります。企業内に技術評価のための専門チームを設置したり、外部の専門家と連携したりするなど、技術デューデリジェンスの重要性が増すでしょう。
5. セキュリティ対策の強化
    - ユーザー生成コードの活用に伴い、セキュリティリスクも高まる可能性があります。企業は、ソフトウェアのセキュリティ監査を強化し、脆弱性への対応を迅速に行う必要があります。また、社員へのセキュリティ教育も重要になってくるでしょう。
6. 外部リソースの活用
    - 社内でのソフトウェア開発だけでなく、外部のユーザー生成コードや、フリーランス開発者の活用も進むと考えられます。必要な機能を外部リソースで迅速に実装することで、ソフトウェア開発の効率化を図ることができるでしょう。
7. 知的財産権の管理
    - ユーザー生成コードや、オープンソースソフトウェアの活用に際しては、知的財産権の管理にも注意が必要です。ライセンスの適切な処理や、自社の知的財産の保護など、法務面でのサポートが重要になってくるでしょう。

民間企業は、ソフトウェア調達の変革を機敏に捉え、新しい技術を積極的に活用していくことが求められます。一方で、品質、セキュリティ、知的財産権などに関するリスク管理も怠ってはなりません。ソフトウェア調達の在り方を見直し、自社に最適な方策を探っていくことが肝要です。社内の技術力を高めつつ、外部リソースも有効に活用しながら、柔軟かつ強靭なソフトウェア調達体制を構築していくことが、企業の競争力を左右すると言えるでしょう。
