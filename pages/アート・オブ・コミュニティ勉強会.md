- → 2冊の本のエッセンスをざっくり掴んでから、広聴AIの実例にあてはめて、自分のコミュニティ観を言葉にする。
3. 自己紹介と今の関心
- → デジタル民主主義やブロードリスニングまわりで、「資源配分を指揮命令できない組織の経営」をずっと気にしてきた話。

1. なぜこの2冊なのか
4. 問題設定：命令できない人たちと成果を出すには？
- → 権限で人を動かせない状況（市民・ボランティア・OSS・コミュニティ）のマネジメント問題としてフレームする。
5. アート・オブ・コミュニティ：社会関係資本を設計する本
- → 「コミュニティ＝社会関係資本の増殖装置」として、構造やワークフローを設計する視点を教えてくれる本。
6. 非営利組織の経営：ミッションと成果で考える本
- → 「ミッション」「誰のための成果か」「ボランティアをどうマネジするか」という非営利の基本フレームを与えてくれる本。

2. フレーム①：コミュニティを構造として見る（AoC）
7. コミュニティ＝社会関係資本のインフラ
- → コミュニティは「仲良しクラブ」ではなく、信頼やつながりという資本を増やすインフラだという整理。
8. コミュニティマネージャ＝“場の構造”を設計する人
- → 人に話しかけるだけでなく、ルール・ツール・ワークフロー・参加の階段を組み立てる役割として再定義。
9. 役割は一枚岩ではない：マイクロスキルの束
- → 「聞く」「翻訳する」「文化を言語化する」「ツールを整える」など、細かい役割の組み合わせとしてコミュニティマネージャを見る。
10. 参加の階段：いきなりコアに連れてこない
- → PR1個・リアクション1つなど、超小さいステップを用意することでコミュニティへの入り口を増やす発想。

3. フレーム②：非営利組織の経営（ドラッカー）
11. ミッションは何か：何をしないかを決める基準
- → 抽象スローガンではなく、「誰にどんな変化を起こすか」を含んだミッションが必要だという話。
12. 誰のための成果か：顧客・受益者をはっきりさせる
- → 市民？行政担当？政治家？研究者？など、「誰の課題を解くのか」を曖昧にしないことの重要性。
13. 成果をどこで測るか
- → お金ではなく、「行動の変化」「能力向上」「意思決定の質」などで成果を測ろうとする視点。
14. ボランティア＝“無給のプロフェッショナル”
- → 単なる善意ではなく、きちんとミッションと成果を共有したうえで仕事を設計する対象として扱うべき、というポイント。

4. ケーススタディ：広聴AIの方向性議論を読み解く
15. 2025-12-06 広聴AIミーティングとは？
- → 参加者とざっくりのテーマを紹介し、「アルゴリズムの会議というよりコミュニティの会議だった」という前提を置く。
16. “広聴AI”には何個の顔があったか？
- → 大量データ俯瞰ツール／熟議支援／市民参加のきっかけ／行政の省力化ツール…など、ミッションが多層化していた状況を整理。
17. レイヤーが混ざっていた問題
- → 「参加感」「全体俯瞰」「深いインサイト」「データ収集設計」など、違うレイヤーの課題が“一つのツール”に押し込まれていた。
18. 典型ユーザ像と現実のズレ
- → 「大量データに困っている前提」で設計されたが、実際には多くの現場はそもそもデータを集められていないというギャップ。
19. アーキテクチャとプラガビリティの悩み
- → 新しいビューや解析手法を取り込めない構成になっていて、「長期安定版としてフリーズするか」「プラガブルに作り直すか」で揺れていた。
20. コアメンバーのモチベーションとボランティア性
- → 「リソースを振り向ける」という表現がズレていて、実際には“本心からやりたいこと”にしか時間を使えないボランティア構造だったこと。
21. 2冊の本で読むと、これは何の問題だったのか
- → AoC的には「コミュニティの構造設計」、ドラッカー的には「ミッション・顧客・成果とボランティアの仕事設計」の問題として見える、というまとめ。

5. 自分のクセと役割の再定義
22. 僕のクセ：人ではなくプロジェクトで覚えてしまう
- → 「Pさん」ではなく「プロジェクトAをやっているPさん」として記憶する／プロジェクトを持ってないQさんがすごく薄くなる、という自己認識を正直に出す。
23. 弱みだけじゃない：アーキテクト型コミュニティマネージャ
- → プロジェクト間の接続や構造が先に見えるのは、「プログラム設計」「ワークフロー設計」には強みになるのでは？という仮説。
24. ドラッカー風に見ると：プログラム・ポートフォリオ担当
- → 個人単位ではなく「どのプログラムにどれだけ力を入れるか」を考える役として、自分の位置づけを再定義してみる。
25. “コミュニティマネージャ苦手”の書き換え例
- タイトル案：
「人のケアは苦手だけど、構造を考えるコミュニティマネージャならやれる」
- → 自分のコミュニティ貢献を「全部やる人」ではなく「この部分専門の人」として描き直すデモ。

6. 参加者ミニワークと締め
26. ワーク①：あなたのコミュニティのミッションを書いてみる
- → 「誰のどんな変化を起こしたいか」を1〜2行で書いてください、という簡単なお題を出す。
27. ワーク②：どのコミュニティマネージャ仕事ならやれそうか
- → AoCで出てきたマイクロスキル（聞く／翻訳／文化を言語化／ツール整備…）の中で「これは自分でもやれそう」を1つ選んでもらう。
28. シェアとクロージング
- → 何人かにミッションと「やれそうな役割」をシェアしてもらい、「命令できない世界をどうマネージするか」を各自の現場に持ち帰ってもらう、で締める。
